## 개요

프론트엔드 채용 공고의 우대 항목 "SSR·ISR·CSR 렌더링 전략 이해"를 파고들며, 세 방식의 본질적 차이와 각 방식이 백엔드·인프라 부하로 어떻게 이어지는지를 정리했다. 특히 ISR의 "낡은 캐시(stale)"가 정확히 무엇을 의미하는지, 그리고 새로고침마다 최신 데이터가 필요한 실시간 대시보드에 ISR이 왜 구조적으로 맞지 않는지를 스스로 질문하며 확인했다. 근거는 Next.js / Vercel 공식 문서와 web.dev 기준.

## 하나의 축: HTML을 언제·어디서 만드는가

세 전략의 유일한 본질적 차이는 **HTML 생성 시점과 위치**다. 나머지 장단점(SEO, 초기 표시 속도, 서버 비용)은 전부 여기서 파생된다.

| 기준 | CSR | SSR | ISR |
|---|---|---|---|
| HTML 생성 위치 | 브라우저 | 서버 | 빌드 서버 → CDN 캐시 |
| 생성 시점 | 런타임(방문 후) | 요청마다 | 빌드 타임 + 주기 재생성 |
| 데이터 신선도 | 매번 최신(클라 fetch) | 항상 최신 | revalidate 주기만큼 지연 |
| 초기 표시 / SEO | 느림 / 불리 | 빠름 / 유리 | 가장 빠름 / 유리 |
| 서버 비용 | 거의 없음 | 요청마다 발생(높음) | 재생성 시에만(낮음) |

## CSR (Client-Side Rendering)

서버는 거의 빈 HTML과 JS 번들만 내려주고, 브라우저가 JS를 실행한 뒤 API에서 데이터를 받아 DOM을 조립한다. 서버는 정적 파일만 서빙하므로 인프라가 가볍고 저렴하다. 대신 초기 HTML이 비어 있어 SEO에 불리하고, 첫 화면까지 번들 다운로드 → 실행 → fetch를 거쳐야 해서 지연이 크다. SPA의 기본 방식.

## SSR (Server-Side Rendering)

요청이 들어올 때마다 서버가 React를 실행하고 데이터를 결합해 완성된 HTML을 만들어 보낸다. 브라우저는 즉시 화면을 그리고(빠른 First Paint, SEO 유리), 이후 JS 번들이 도착하면 이미 그려진 HTML에 이벤트를 연결하는 **하이드레이션(hydration)** 이 일어난다.

- **백엔드/인프라 연결**: 요청마다 서버가 렌더링 연산을 하므로 CPU 비용이 발생한다. Node.js는 단일 스레드라 무거운 렌더링이 이벤트 루프를 막을 수 있고, 서버리스에 배포하면 콜드 스타트가 초기 지연으로 직결된다.

## ISR (Incremental Static Regeneration)

ISR은 CSR·SSR과 나란한 '네 번째 렌더링 방식'이 아니라, **SSG(정적 생성) 위에 얹은 캐싱 전략**에 가깝다. 페이지는 여전히 정적 HTML로 생성되고, 거기에 지능적인 캐시 무효화를 더한 것이다.

빌드 타임에 정적 HTML을 만들어 CDN 캐시에 올려두고, 사이트를 전체 재빌드하지 않고도 페이지 단위로 갱신한다. 이 방식으로 정적의 이점을 유지하면서 수백만 페이지까지 확장할 수 있다.

### stale-while-revalidate — "낡은 캐시"의 정확한 의미

가장 중요한 개념. `revalidate: 60`을 설정하면 그 60초 동안 모든 방문자가 동일하게 생성된 버전을 보게 되고, 캐시를 무효화하는 유일한 방법은 60초가 지난 뒤 누군가 그 페이지를 방문하는 것이다.

핵심은 **누구도 대기하지 않는다**는 점이다. 갱신 시간이 지난 뒤 방문한 사람도 여전히 캐시된(낡은) 버전을 즉시 받아 빠르게 응답을 얻고, 동시에 백그라운드에서 재생성이 트리거된다. 새 버전이 성공적으로 생성되면 캐시된 버전을 교체하고, 그 다음 방문자부터 갱신된 콘텐츠를 받는다. 재생성이 실패해도 이전에 성공적으로 생성된 페이지가 그대로 유지되고, 다음 요청에서 다시 시도한다.

여기서 "낡은 캐시 = 이전 빌드 버전"이라는 건 **처음 한 번만** 맞다. 한 번이라도 재생성이 일어나면 캐시에 있는 건 빌드 산출물이 아니라 **런타임에 백그라운드에서 새로 구워진 최신 스냅샷**으로 교체된다. 정확한 표현은 "마지막으로 성공적으로 (재)생성된 시점의 데이터 스냅샷"이다.

캐시 동작은 응답 헤더 `x-nextjs-cache`로 관찰할 수 있다: `HIT`(캐시 서빙), `STALE`(캐시 서빙 + 백그라운드 재검증), `MISS`(캐시에 없어 새로 렌더), `REVALIDATED`(온디맨드 재검증으로 재생성).

## 실시간 대시보드에 ISR을 쓸 수 있나 → 구조적으로 부적합

"새로고침을 누를 때마다 최신 데이터"라는 요구는 ISR로 만족시킬 수 없다. 두 가지 이유가 걸린다.

1. **ISR 캐시는 모든 방문자가 공유하는 캐시다.** revalidate 주기 동안 모두가 같은 버전을 본다. "이 사용자의, 지금 이 순간 데이터"인 대시보드와 정면충돌한다.
2. **새로고침(F5)이 최신을 보장하지 않는다.** 새로고침은 같은 URL을 CDN에 다시 요청하는 것뿐이고, 캐시가 fresh면 `HIT`으로 방금 본 것과 똑같이, stale이면 낡은 캐시를 즉시 주고 재생성은 뒤에서만 돈다. 백그라운드 재생성의 열매는 **다음 방문자**가 받는다. 즉 "새로고침한 그 사람"은 최신을 받지 못한다.

공식 가이드도 실시간 업데이트가 필요한 경우(주식 시세, 실시간 스코어 등)에는 클라이언트 사이드 페칭이나 SSR을 쓰라고 안내하며, 실시간 대시보드·사용자별 페이지는 SSR이나 CSR을 써야 한다고 명시한다.

### 그럼 실시간 대시보드는? — 데이터 페칭 전략의 문제

이건 렌더링 전략이 아니라 **데이터 페칭 전략**의 영역이다.

- **클라이언트 사이드 페칭(가장 일반적)**: React Query / SWR로 브라우저가 직접 API를 호출. `refetchInterval`로 폴링, 갱신 버튼은 `refetch()`. ISR 캐시와 무관하게 API를 직접 때리므로 "누른 사람이 즉시 최신"이 성립한다. 진짜 초 단위 실시간이면 폴링 대신 WebSocket / SSE로 서버 푸시.
- **하이브리드(실무 정답)**: 페이지 껍데기(레이아웃·정적 UI)는 SSG/ISR로 빠르게 띄우고, 그 안의 살아 있는 데이터만 React Query로 클라이언트에서 가져온다. 로그인 뒤에서만 보이는 대시보드는 SEO가 불필요하므로 껍데기까지 CSR로 두는 경우도 많다.

```js
const { data, refetch } = useQuery({
  queryKey: ['dashboard'],
  queryFn: fetchDashboard,
  refetchInterval: 5000, // 5초 폴링
});
// 갱신 버튼: onClick={() => refetch()} → 즉시 최신
```

## 인프라 관점 정리

ISR의 강점은 **CDN 엣지의 공유 캐시**에서 나온다. 미리 만든 정적 HTML을 엣지에서 뿌리고 원본 함수는 재생성할 때만 돌기 때문에 서버 부하·비용이 낮다. 반대로 실시간 대시보드는 정의상 "매번 원본의 최신 상태"를 요구하므로 캐시가 계속 무효화·우회되어 캐시 이점이 0에 수렴한다 — 캐싱 전략인 ISR을 얹는 것 자체가 도구 선택 오류.

셀프 호스팅/다중 인스턴스 주의점: 기본 파일시스템 캐시는 인스턴스별로 분리되어, 인스턴스 간 조율을 위해서는 공유 커스텀 캐시 핸들러가 필요하다. 쿠버네티스에서 파드마다 캐시가 따로 놀면 "분명 갱신했는데 어떤 사용자는 옛날 걸 본다"는 문제로 이어진다.

## Q&A 하이라이트

- **Q. ISR의 낡은 캐시는 이전 빌드 버전인가?** → 최초 1회만 빌드 버전. 재생성 이후로는 "마지막으로 성공 재생성된 최신 스냅샷"으로 교체된다.
- **Q. 실시간 대시보드를 ISR로, 새로고침마다 최신 받기 가능한가?** → 불가능. ISR 캐시는 공유 캐시이고 새로고침이 최신을 보장하지 않는다. 클라이언트 페칭(폴링/WebSocket) 또는 SSR의 몫.
